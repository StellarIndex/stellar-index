# Supply from ClickHouse — baseline snapshot + forward flows

**Status: design.** How to serve circulating/total supply for **every token**
when the served tier is (re)built from the ClickHouse Tier-1 lake (ADR-0034),
given the lake captures *activity* (events/ops) but not full ledger-entry
*state*. The answer: supply is a mint/burn calculation, and the flows we need
are in the lake for the common cases; the rest is closed with one cheap
baseline snapshot — NOT the full state-capture.

## 1. Supply is a flow, not (necessarily) a state scan

`supply(asset, t) = baseline(asset, t0) + Σ mints(t0..t) − Σ burns(t0..t)`

If we have a starting balance and every issuance/destruction flow afterward,
we never need to scan full state. Whether the flows are in the lake depends on
the token class.

## 2. By token class

| class | supply formula | flows in the lake? |
|---|---|---|
| **SEP-41 / Soroban tokens** | Σ`mint` − Σ`burn` − Σ`clawback` from contract events | **✅ fully** — `contract_events`, all history. `baseline = 0` at contract genesis. |
| **Classic issued assets (ALL history)** | Σ`mint` − Σ`burn` (CAP-67 unified asset events) | **✅ fully** — see §2a. Our galexie archive is uniformly V4 meta, so stellar-core synthesized CAP-67 `mint`/`burn`/`transfer` events for classic movements back to **genesis** (verified: `mint`/`burn` present at ledger 25M / 2019). `baseline = 0` at asset issuance. |
| **XLM (native)** | total = ledger header `total_coins`; circulating = total − locked/non-circulating account balances | total **✅ `ledgers`** (exact, per ledger); circulating needs the locked-account set (small curated list + their balances). |

## 2a. Why classic supply is in the lake for ALL history (not just ≥P23)

`ingest.GetTransactionEvents()` returns events only for V3 (Soroban) or **V4**
meta. Our archive was generated by modern stellar-core replaying all history,
which writes **V4 meta for every ledger** and emits the CAP-67 token-transfer
events (`transfer`/`mint`/`burn`/`clawback`) for classic asset movements — even
for 2015–2024 ledgers that originally closed under pre-CAP-67 protocols. So
`contract_events` already carries every classic asset issuance/destruction from
genesis. The hash-chain verification (0 breaks) confirms the archive is
authentic history; the V4 container is just the modern meta format.

Consequence: classic-asset supply does **not** need `ledger_entry_changes` or a
baseline snapshot after all — it's `Σmint − Σburn` over `contract_events` for the
asset's SAC contract, full history. (A baseline snapshot remains a useful
*cross-check* and a shortcut to avoid summing billions of events, but it is no
longer required for correctness.)

## 3. The cheap closure: one baseline snapshot + forward flows

For classic assets that predate P23 (most major ones), don't reconstruct
pre-P23 history. Instead:

1. **Take a one-time baseline** of each asset's circulating supply at a cut
   ledger `t0` (choose P23 activation, or simply "now"):
   - source: the **live Postgres supply tier** we already maintain (it has
     current classic supply), or a one-time **bucket-list / checkpoint state
     read** (sum of non-issuer `TrustLineEntry` balances). Either is a single
     read, not a per-ledger walk.
2. **Roll forward from `t0` using lake flows**: CAP-67 `mint`/`burn` events for
   classic assets (≥ P23), SEP-41 `mint`/`burn`/`clawback` for Soroban tokens.
3. **XLM**: `total_coins` from the header (exact, per ledger); circulating =
   total − a curated locked-account set whose balances come from the same
   baseline snapshot rolled forward by the relevant account-debit/credit flows
   (or simply refreshed from the live tier).

This gives **current + forward supply for every token from the lake + one
snapshot**, with no dependency on `ledger_entry_changes` history.

## 4. What still needs full state (deferred)

Supply does **not** — §2a removed that dependency. What remains full-state /
full-explorer work (parked):

- **Arbitrary account balances + contract data/state** for explorer entity
  pages (any account's holdings on any date) — needs `ledger_entry_changes`
  and/or current-state materialization. Not a supply requirement.
- **XLM circulating** needs the locked/non-circulating account set's balances.
  That set is small + curated, and its balance moves are themselves CAP-67
  `transfer` events, so it's derivable from a one-time baseline + transfer
  flows — no full-state scan required.

## 5. Migration-plan correction

ADR-0034 plan §10a ("drop + rebuild supply tables from CH") is now achievable
**from CH for every token**:
- **SEP-41 + classic-asset supply** → rebuild from CH (`Σmint − Σburn` over
  `contract_events`, full history — §2a). ✓
- **XLM** → total from `ledgers.total_coins`; circulating = total − locked-set
  balances (baseline + CAP-67 transfer flows). ✓
- Reconcile the CH-rebuilt numbers against the live tier at cutover before
  dropping the live path.

## 6. Build outline (when scheduled)

- `internal/supply` already has the three algorithms (XLM / classic / SEP-41).
- Add a `baseline(asset, ledger)` reader (from live tier or checkpoint state).
- Point the SEP-41 + CAP-67 mint/burn observers at `clickhouse.StreamContractEvents`
  (the Phase-4 adapter) instead of the live dispatcher for re-derivation.
- Reconcile the rolled-forward number against the live tier at cutover (they
  must agree at `t0`), then serve.
