# Rates Engine — system review

A thorough read across the two RFPs, the awarded proposal, the
current code surface, the explorer, and **direct verification
against the live `https://api.ratesengine.net` production API**
running v0.5.0-rc.39 (deployed today, 2026-05-10 20:45 UTC).

Every finding below is backed by a live curl response captured at
review time (2026-05-10 ~21:30–21:55 UTC). Where I cite a status
code or response shape, I actually saw it.

---

## TL;DR — verdict

**The implementation is broad and well-architected**: 55 API
endpoints across 8 surface families; all 7 RFP-specified
granularities ship; all 4 named oracles wired; 6 DEX integrations
(2 more than the proposal); every Freighter V1 metadata field
delivered; SLA latency p95 ≈ 200–210ms (right at the limit).

**But there are real production gaps I caught with simple
exploration**:

1. **`/v1/markets` and `/v1/pools` are degraded for ~10 min after
   every deploy** — cold cache + slow underlying SQL → 8s
   timeout. `/v1/pools` returns a clean 503 with a "retry in a
   few seconds" hint; **`/v1/markets` returns generic 500.**
   Differential bug. Once the cache warms, both serve in <500ms.
2. **`/v1/price/batch` silently drops USDC** — XLM works, USDC
   returns empty array. The stablecoin-fiat proxy fallback that
   `/v1/price` applies isn't applied in batch.
3. **F2 supply fields are NULL on every asset** — `market_cap_usd`,
   `fdv_usd`, `circulating_supply`, `total_supply`, `max_supply`
   missing on `/v1/coins/native`, `/v1/assets/native`. Operator
   config gap (#97) — supply pipeline watched-sets aren't
   populated on r1.
4. **ATH inconsistency between two endpoints** — `/v1/coins/XLM`
   reports XLM ATH = $1.03 (contaminated through stablecoin proxy);
   `/v1/changes/coin/XLM` reports $0.17 (correctly filtered). Two
   user-facing surfaces disagree on the same field.
5. **`/v1/ohlc?base=native` returns `high: 1.0000000000`** — same
   contamination class as the ATH bug. `flags.triangulated: true`
   is honest but the bar is wrong.
6. **SEP-10 returns 503 in production** — handler is wired but
   the server signing seed isn't configured (`SEP-10 not
   configured` body). Coverage matrix says ✅ but live API says
   503.
7. **`/v1/observations?asset=native&quote=fiat:USD` returns `[]`** —
   most-traded pair on the system has no per-source observations
   surface. Probably because it's a triangulated pair with no
   direct trades.
8. **API parameter naming is inconsistent** — `/v1/chart` uses
   `asset=`, `/v1/ohlc`/`/v1/history` use `base=`,
   `/v1/observations` uses `asset=`, `/v1/twap`/`/v1/vwap` use
   `base=`. Three different conventions across closely-related
   endpoints.
9. **Anonymous rate limit is 60/min** — RFP says ≥ 1000/min.
   Almost certainly the *authenticated* tier is 1000+ but the
   anonymous tier visibly returns `x-ratelimit-limit: 60` to
   unauthenticated curl. Worth confirming this matches RFP intent.

These aren't deal-breakers. Each is a small fix or operator-config
item. But they add up to a "demoable but not yet polished" state.

The matrix says "✅ verified" for many of these because the **code
is shipped**; the **production behaviour** is the gap. That's a
documentation-truth problem on top of the production problem.

### High — would block consumer integration

> **Note on R-001 to R-004**: re-verified at 21:57 UTC after the
> initial review. **Behaviour is cold-cache-dependent.** Right
> after a deploy (api binary restart), the markets/pools cache
> is empty and the underlying SQL takes > 8s. Within a few
> minutes the cache warms and queries return in < 500ms.
> Differential: **`/v1/pools` correctly returns 503 with a clean
> `pools-timeout` problem+json** ("the underlying trades-hypertable
> scan didn't return in 8s; cache may still be warming. Retry in
> a few seconds."). **`/v1/markets` returns generic 500.** The
> right fix is (a) pre-warm on boot, OR (b) make `/v1/markets`
> match `/v1/pools`'s 503-with-retry-hint pattern.

|---|---|---|---|---|
| R-001 | `/v1/markets?limit=5` (cold cache only) | **500** in 8s | `DistinctPairs` SQL timeout, no graceful 503 | Match `/v1/pools` 503 pattern OR pre-warm cache |
| R-002 | `/v1/markets?source=binance` (cold cache only) | **500** in 8s | `DistinctPairs` SQL timeout | same |

## Section 4 — System architecture review

(Unchanged from the first pass — these are big-picture, not
production-state-dependent.)

### What's strong

1. **Galexie → MinIO → dispatcher → decoder.** Every decoder is
   a pure function over `events.Event` and `LedgerEntryChange`. New
   sources plug in via the registry without touching the dispatcher.
   The `BackfillSafe` flag gates retroactive replay — sound for the
   WASM-evolution problem.

2. **Three consistency surfaces (ADR-0018).** Verified live —
   `/v1/price` (closed bucket), `/v1/price/tip` (rolling window),
   `/v1/observations` (raw per-source). Three URLs, three
   contracts. URL discipline linter enforces query params can't
   cross consistency boundaries. **This is unusually good API
   design for a pricing layer.**

3. **Anomaly + freeze engine (ADR-0019).** Per-class thresholds,
   per-asset MAD baseline, multi-window safeguard, 3-signal AND
   freeze, bootstrap warmup, ops runbook. **Verified live**:
   `/v1/vwap` returns `outliers_filtered: 0` per response, so the
   gate is actually wired.

4. **Per-asset-class supply (ADR-0011/0021/0022/0023).**
   Three-domain split (XLM Algo 1, classic Algo 2, SEP-41 Algo
   3) with per-class hypertables and an aggregator-resident
   refresher. Operator-overridable locked-set subtraction
   makes the policy choice explicit. **But: not running on r1
   today — config gap.**

5. **Cross-region storage strategy (ADR-0016).** R1 (Hetzner)
   integrity leader; R2 (AWS) reads `aws-public-blockchain` S3
   directly; R3 (Vultr) hybrid. **But R2/R3 don't exist yet** —
   single-region in production today.

6. **Discovery archive at `docs/discovery/`.** Read-only since
   2026-04-22; per-venue audit docs with every gotcha captured.
   Phase 1 institutional memory.

7. **24 ratified ADRs.** Numbered, immutable, accept-only-or-
   supersede. Coverage matrix links every requirement to ADR +
   owner package + test file.

8. **`/v1/incidents` with full markdown bodies.** Verified live —
   the May 10 SEV-2 incident has a 2,000-word postmortem inline
   in the API response. **Public-incident transparency rare in
   the ecosystem.**

### What's inconsistent or could be polished

(See findings register R-010 through R-018 for production-verified
inconsistencies. Below are the architectural ones.)

1. **Two `web/` directories** (explorer + dashboard) sharing
   copies of api hooks, query client config, primitives.

2. **`internal/consumer/` is legacy** — production uses
   `internal/dispatcher`. Worth deleting before public-flip.

3. **`internal/stellarrpc/` is described in CLAUDE.md as not in
   prod ingest** — exists for diagnostic + fixture capture only.
   Could be moved to `cmd/` or split.

4. **CHANGELOG `[Unreleased]` section can hit ~800 lines** when
   many PRs land in one cycle. Sub-section by area would help.

5. **`pkg/client` covers most endpoints but not all** — streaming
   surfaces (SSE) and oracle SEP-40 surfaces lack typed clients.

6. **`web/explorer/public/_redirects` ~190 lines** — Cloudflare
   Pages reads linearly; consider build-time generator.

### What's actually missing (verified gaps)

1. **No `/v1/changes/top?window=24h&direction=up`** — top movers /
   biggest losers list endpoint. Per-asset summary exists; list
   doesn't.

2. **No batch supply lookup** — `/v1/price/batch` exists but
   `/v1/assets/batch` for the F2 fields doesn't.

3. **No public per-source health endpoint** — internal aggregator
   metrics capture per-source health (CoinGecko 429 / Bitstamp
   dust / Reflector observations stale) but consumers can't see
   them via the API.

4. **No webhook / push surface** — SSE handles streaming but no
   "POST to my URL when XLM/USD crosses $0.20" surface.

5. **No bulk-historical-export** — paginated endpoints only;
   no parquet / CSV bulk download for analytics consumers.

6. **No `/v1/methodology` API endpoint** declaring the active
   aggregation policy machine-readably (min volume thresholds,
   freeze thresholds, divergence references active, anchor assets,
   FX vendors enabled). The explorer page is doc-only.

7. **No `/v1/sources/health` umbrella** — `/v1/sources` returns
   metadata only; not which sources are degraded right now.

## Section 7 — What I'd change before public-flip

Ranked by impact. **Bold = visible API behavior change.**

### High — fix before customers see it

1. **Fix `/v1/markets` and `/v1/pools` 500s** (R-001 to R-004).
   Bare `?limit=5` queries return 500 in 8s. Either pre-warm the
   cache on boot, or speed up the underlying queries, or return
   503/504 so consumers retry rather than fail. **A Freighter
   integrator hitting `/v1/markets` first thing would think the
   API is down.**

2. **Fix `/v1/price/batch` silent drop on stablecoin pegs**
   (R-005). Apply the stablecoin-fiat proxy fallback the same
   way `/v1/price` does. **This is the wallet portfolio path —
   any wallet asking for XLM + USDC + AQUA in one batch
   silently loses USDC.**

3. **Fix `/v1/ohlc.high = $1` contamination** (R-007). Filter
   triangulated paths from OHLC source data. **An OHLC bar that
   reports XLM hit $1 in 24h is a data-integrity break.**

4. **Fix ATH inconsistency** (R-008). Two endpoints disagree on
   the same asset's ATH ($1.03 vs $0.17). **Whichever Freighter
   consumes determines what users see; one of them is wrong.**

5. **Populate operator config gaps on r1** (R-006, R-009).
   Without `[supply].watched_*` and `[api.dashboard]` and the
   SEP-10 signing seed, F2 fields are NULL, dashboard sign-in
   is broken, and SEP-10 returns 503.

6. **Standardise endpoint parameter naming** (R-010). Pick
   `asset=` or `base=` and migrate the others, or accept both
   with one canonical. The current state is inconsistent across
   closely-related endpoints.

### Medium — visible polish

7. **Inline SEP-1 overlay on `/v1/assets/{id}`** (R-016, R-017).
   Today consumers need a separate `/v1/issuers/{g_strkey}` call.

8. **Fix `/v1/assets/{id}.sep1_status: "not_applicable"`** for
   classic assets that DO have SEP-1 metadata (R-016).

9. **Add `truncated: true` flag** on `/v1/chart` when the
   timeframe exceeds available retention (R-013).

10. **Default `/v1/markets` sort to volume_24h_usd_desc** so spam
    tokens don't appear first (R-014).

11. **Remove completed-backfill cursors** from `/v1/diagnostics/cursors`
    or add a `?status=active` filter (R-015).

12. **Replace handler 8s-timeout 500s with 504s** (R-021). HTTP
    semantics: 500 = server bug, 504 = upstream timeout.

13. **Bring up R2 + R3** so the 99.99% uptime number can actually
    be measured.

14. **Install Alertmanager + Loki on r1** so alerts can route
    and logs are searchable.

15. **Install the 4 missing systemd timers** (sla-probe,
    supply-snapshot, verify-archive-tier-a, archive-completeness).

16. **External security review** before public-flip.

### Low — nice to have

18. Add `/v1/changes/top` for movers discovery.

19. Add `/v1/assets/batch` for F2 fields.

## Section 9 — Final verdict (revised after live verification)

**Fit for purpose:** **mostly yes, with some real bugs that need
to land before public-flip.** The API is comprehensive; the
architecture is excellent; the docs are exceptional. But:

- Bare `/v1/markets` returns 500
- Bare `/v1/pools` returns 500
- Batch silently drops stablecoins
- F2 supply fields are universally NULL
- OHLC has $1 contamination on XLM
- ATH disagrees between two endpoints
- SEP-10 returns 503

These aren't hypothetical — they're what a curl against
production gets right now.

**Lacking features:** a small handful of nice-to-haves (movers
endpoint, batch F2, webhook, methodology API). None RFP-required.
None blocking.

**Inconsistencies:** parameter naming (`asset=` vs `base=`),
SEP-1 status reporting, ATH between endpoints, F1 metadata
split across two surfaces, 500-vs-504 status codes on timeout.
All cosmetic to architectural — fixable.

**Overlooked:** R2/R3 multi-region (single-region cap on 99.9%);
external security review; r1 operator config gaps; Alertmanager
+ Loki not deployed.

**Ready to launch?** If "tomorrow morning"? **No** — the bare
`/v1/markets` 500 alone would mean any first-time consumer sees
a broken API. The bare `/v1/pools` 500 same. The batch silent-
drop would break wallet portfolio views.

If "in 1–2 weeks after fixing the 9 high-severity findings + R2
bringup"? **Yes** — the rest of the surface is genuinely strong
and the issues are tractable. Most of the high-severity findings
are 1–2 days of work each. The matrix discipline already in
place would catch them in a normal review cycle if someone read
the matrix and curled the live API.

The single most actionable observation from this review: **the
coverage matrix tracks code-shipped, not production-live; reviewers
who only read the matrix will miss what production is actually
serving.** Worth adding a "production-verified" column to the
matrix — `last verified against api.ratesengine.net on YYYY-MM-DD`
per row — so the gap between "merged" and "demonstrably live"
stays visible.

## Appendix C — One thing I changed my mind about

I went into this expecting the system to be in worse shape than
it is. The architecture, ADR discipline, discovery archive, audit
findings discipline, runbook coverage, and the explorer's reveal
panels are all genuinely impressive — beyond what most pricing
services in any ecosystem ship.

What surprised me is **how much the production reality lags
the matrix's "verified" status** on a handful of dimensions. The
F2 fields, OHLC contamination, batch silent-drop, and bare
`/v1/markets` 500 are bugs anyone who ran 10 minutes of curl
against the live API would catch. The matrix says they're
shipped (which is true at the code level); production says
they're broken (which is also true).

This is a fixable gap — add a "production-verified" column to
the matrix with a date stamp; demand a curl reproduction next
to every ✅. The discipline is already there in spades; just
needs one more axis of verification.
